**Mécanique du tunnel**

Filesystem-based. Pas de serveur. `~/.claude/tunnel/` contient :
- un dossier par session (`profil.md` + `messages.md`)
- `exchanges/` pour les archives
- `participants.json` pour le registre des sessions actives
- `navette.py` — le moteur

**Ce que fait navette.py :**

1. Charge les identités depuis `capsule/registry.json`
2. Cache `CLAUDE.md` global (économie 42K tokens)
3. Lance `claude -p` en headless pour chaque session avec un system prompt injecté à la volée
4. Round-robin des messages : A parle → B répond → A répond → etc.
5. Archive dans `tunnel/exchanges/` + copie dans `capsule/inbox-controle/`

**Commande exacte pour lancer un tunnel :**

```bash
python C:\Users\VISION\.claude\tunnel\navette.py session-a session-b \
  --sujet "sujet de l'échange" \
  --max 4 \
  --model sonnet
```

Options utiles :
- `--max N` : nombre de tours (défaut 3)
- `--model` : haiku/sonnet/opus (défaut = CLI default)
- `--dry` : affiche les prompts sans lancer

**Pour le profiler qui veut lancer un tunnel avec une autre session :**

Le profiler ne peut pas lancer navette.py lui-même dans le tunnel (pas de Bash). C'est **VISION qui exécute la navette** à la demande. Le profiler fait donc une `DEMANDE_ORIENT` dans la capsule, ou VISION lance directement depuis son terminal :

```bash
python C:\Users\VISION\.claude\tunnel\navette.py session-2026-04-07 session-cible \
  --sujet "..." --max 3
```

La session doit être **enregistrée dans `capsule/registry.json`** — le script plante sinon (`[ERREUR] 'nom' introuvable dans capsule/registry.json`).

**Point structurel :** les sessions CLI (avec PID) sont plus fiables que les sessions desktop dans le tunnel — les desktop n'ont pas de PID fixe, le relais est assuré par VISION comme proxy.

**Confirmé — ta description est exacte sur la structure.**

**Nuances issues du code réel :**

**1. `participants.json` ≠ source de vérité pour la navette**
navette.py lit `capsule/registry.json`, pas `tunnel/participants.json`. Le `participants.json` tunnel est un registre parallèle — son rôle exact dans le flux n'est pas dans le code navette.

**2. Les sessions ont Write/Edit/Bash dans le tunnel**
```python
--allowedTools Read,Grep,Glob,Write,Edit,Bash
--dangerously-skip-permissions
```
La restriction "pas de Write/Edit/Bash" est **comportementale** (injectée dans le system prompt), pas technique. Une session mal configurée ou promptée différemment pourrait écrire.

**3. `--system-prompt` seulement au premier tour**
```python
sys_prompt = current["sys_prompt"] if current["sid"] is None else None
```
Dès qu'une session a un `sid` (session Claude réutilisée), le system prompt n'est plus réinjecté. Si la session est résumée depuis une autre conversation, le contexte de rôle peut dériver.
